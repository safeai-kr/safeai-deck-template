As-is · 현행 구조

지금 구조가 아픈 세 지점

차트 동기화 자체는 동작한다. 문제는 상태가 한 곳에 뭉쳐 있고, 이벤트 처리가 흩어져 있다는 것 — 기능을 늘릴수록 비싸지는 구조다.

문제 1
단일 persistConfig store
서버 데이터와 UI 인터랙션 상태가 한 store에 섞여 영속화·리렌더 경계가 뒤엉킨다.
문제 2
currentEvent if-사슬
이벤트 분기가 파일 3곳에 흩어져, 새 이벤트를 추가하면 세 군데를 고치고 하나만 빠뜨려도 조용히 통과한다.
문제 3
환자 전환 시 stale
환자를 바꿔도 이전 progressIndex · currentEvent가 남아, 새 차트에 잘못된 커서·가이드가 그려진다.

03
개선 1

Store를 둘로 가른다

데이터와 인터랙션은 수명도, 변경 빈도도, 영속화 정책도 다르다. 하나의 persistConfig에 묶으면 둘 다 손해다.

ChartInteractionStore
UI 커서와 이벤트 상태. 자주 바뀌고, 환자마다 초기화되어야 한다.
progressIndex · 시간 커서(number)
currentEvent · { name }
reset() 보유
ChartDataStore
서버에서 받은 바이너리·측정 데이터. 무겁고 안정적이며 IndexedDB에 영속화된다.
receiveData
IndexedDB 영속화
reset() 대상 아님
04
개선 2

이벤트 처리를 한 곳에 모은다

if-사슬 3파일 → 맵 1곳
// 이벤트 → 핸들러 맵 (한 파일)
const guideHandlers: Record<GuideEventType, Fn> = {
  pWave: (ctx) => { … },
  qrsPeak: (ctx) => { … },
  camArrow: (ctx) => { … },
};

// reaction 1개가 전부를 디스패치
reaction(() => currentEvent.name,
  (name) => guideHandlers[name](ctx));
추가 지점 3곳 → 1곳
새 이벤트는 맵에 한 줄만 넣으면 끝.
누락 = 컴파일 에러
유니온에 이벤트를 더하면 핸들러가 없을 때 타입이 막는다.
추적 지점이 하나
"이 이벤트 어디서 처리되지?"의 답이 항상 같은 파일.
05
개선 3

stale는 한 줄로, 페이로드는 최소로

환자 전환 = reset() 한 줄
인터랙션 상태를 store가 소유하므로, 전환 시점에 한 번만 비우면 stale가 사라진다.
// 환자 바뀔 때
interactionStore.reset();
// progressIndex, currentEvent 초기화
데이터 store는 건드리지 않는다 — 캐시 유지.
currentEvent는 { name }만
UI row를 통째로 넘기지 않는다. 핸들러가 필요한 데이터는 store에서 직접 읽는다.
As-is
UI row 통째
To-be
{ name }
직렬화·리렌더 비용이 페이로드에 비례해 줄어든다.

07
정리하면

경계를 나누니, 안전과 명확이 따라온다

유지보수성
이벤트 추가 지점이 3곳 → 1곳. 고칠 곳이 줄었다.
타입 안전
핸들러 누락이 런타임 버그가 아닌 컴파일 에러가 된다.
디버깅 명확
store 경계와 '별도 채널'이 문서화되어 상태 추적이 쉽다.
