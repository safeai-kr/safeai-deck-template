02
왜 계약을 먼저?

이유는 하나 — 바꾸는 비용

계약(문서)에서 고치는 건 공짜지만, 코드와 DB로 다 만든 뒤에 고치는 건 비싸고 위험하다. 그래서 명세를 바로 코드로 옮기지 않고, 값싼 단계에서 결정을 끝낸다.

계약에서 고치기 = 공짜
문장 한 줄, 필드 하나 바꾸면 끝. 아직 아무것도 만들지 않았으니 되돌릴 게 없다.
코드·DB 고치기 = 비쌈
엔드포인트·스키마·마이그레이션·테스트가 다 엮여 있다. 하나 바꾸면 연쇄로 무너진다.
03
1단계

리소스와 행위 뽑기

명세를 읽으며 · 명사에 밑줄, 동사에 동그라미

"사용자는 할 일을 등록하고, 목록을 조회하고, 완료 표시할 수 있다."

명사 → 리소스
할 일(Todo) — 저장하고 주고받을 데이터의 정체
동사 → 행위(엔드포인트)
등록 · 조회 · 완료표시 — 각각이 API 하나. "3개쯤 필요하겠구나"
04
2단계

데이터 모델 정의

리소스의 "모양" — 무엇을 저장하고 돌려줄지
Todo = {
id       고유번호
title     할 일 내용
completed 완료 여부 (true/false)
createdAt 생성 시각
}
왜 계약보다 먼저(또는 같이)?

API는 결국 데이터를 주고받는 통로다.
그래서 데이터 모양이 안 정해지면 통로(계약) 모양도 못 정한다.

05
3단계 · 제일 중요

API 계약 정의

각 행위를 4가지로 못 박는다
메서드 + 경로
어떻게 부를지
입력 (요청)
무엇을 받나 — body·파라미터
출력 (성공)
상태코드 + 돌려주는 모양
실패 (에러) ★
어떤 상황에서 어떤 에러 — 초보자가 놓치는 칸
Todo 로 채우면
POST/todos
입력 { title } · 성공 201 { id, title, completed:false } · 실패 400 (title 없음)
GET/todos
입력 없음 · 성공 200 [ {…}, {…} ]
PATCH/todos/:id
입력 { completed:true } · 성공 200 { … } · 실패 404 (없는 id)
06
4단계

구현 — 엔드포인트는 세 겹

코드는 계약을 "참으로 만드는" 작업일 뿐. 엔드포인트 하나를 항상 아래 세 겹으로 짠다.

01
입력 검증
요청이 계약에 맞나? title이 비었으면 여기서 400.
02
비즈니스 로직
실제 해야 할 일. "완료로 바꾼다", "새 항목을 만든다".
03
데이터 접근
DB에 읽고 쓴다. 그리고 결과를 계약의 응답 모양으로 변환해 반환.
07
5단계

테스트 = 계약을 그대로 옮기기

새로 뭘 생각하는 게 아니다. 3단계 계약에 적어둔 성공·실패 케이스를 그대로 테스트로 옮기면, 계약이 잘 적혀 있는 만큼 테스트 목록이 저절로 나온다.

성공 케이스
POST /todos 에 title 넣으면 → 201 과 completed:false
GET /todos → 200 과 배열
실패 케이스
title 없이 보내면 → 400 나오나?
없는 id로 PATCH → 404 나오나?
08
기억 트리거

한 줄로 — 명세를 계약으로 번역하고,
계약을 코드로 구현한다

큰 원리명세→코드 사이에 계약을 먼저 세운다 · 이유 = 바꾸는 비용
1 · 2명사=데이터 / 동사=엔드포인트 개수 · 데이터 모양이 통로 모양의 전제
3 (핵심)메서드+경로 / 입력 / 성공 / 실패 — 에러 칸을 반드시 여기서 채운다
4 · 5구현=검증→로직→데이터접근 3겹 · 테스트=계약의 성공/실패 목록

주니어에게는 "에러 케이스를 계약 단계에서 못 박는다"를 강조 — 정상 흐름만 짜다가 터지는 사고를 막는다.
